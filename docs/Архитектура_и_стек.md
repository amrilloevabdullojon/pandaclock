## 1. Технический стек (рекомендация)

### 1.1 Принципы выбора стека

1. **Один язык на всём стеке** — TypeScript везде (web, mobile, backend). Одна команда может работать со всеми частями.
2. **Портируемость** — никакой жёсткой привязки к Vercel, чтобы потом мигрировать на узбекские серверы без переписывания.
3. **Зрелые технологии** — никакой экзотики, проверенные решения с большим комьюнити.
4. **Multi-tenant с первого дня** — все решения учитывают изоляцию данных.

### 1.2 Frontend

#### Web (для руководителей, HR, админов)
- **Next.js 15+** (App Router, React Server Components)
- **TypeScript**
- **TailwindCSS** — стили
- **shadcn/ui** — компонентная библиотека (копируется в проект, не зависимость)
- **Zustand** — лёгкий state management
- **TanStack Query (React Query)** — работа с API
- **react-hook-form + zod** — формы и валидация
- **Recharts / Tremor** — графики для дашбордов
- **next-intl** — мультиязычность

**Почему Next.js:**
- Идеально работает на Vercel (наша MVP-площадка)
- Легко мигрируется на любой Node.js хостинг (Docker)
- SSR/SSG для SEO лендинга
- API routes для простых эндпоинтов

#### Mobile (для сотрудников)
- **React Native + Expo** (SDK 52+)
- **TypeScript**
- **Expo Router** — навигация
- **NativeWind** — Tailwind для React Native
- **React Query** — те же паттерны что в web
- **Expo Notifications** — push (FCM + APN)
- **expo-location** — геолокация при отметке
- **expo-camera + expo-face-detector** — биометрия / фото при отметке

**Почему React Native + Expo:**
- Один кодбейз для iOS и Android
- Команда из 1-2 человек справится с мобильным
- Expo упрощает CI/CD и push-уведомления
- Можно переиспользовать React-знания фронтенд-разработчиков

### 1.3 Backend

#### Основной API
- **Node.js 22+ (LTS)**
- **NestJS** — фреймворк (модульная архитектура, готова для микросервисов)
- **TypeScript**
- **Prisma ORM** — типобезопасная работа с БД
- **Zod** — валидация входных данных
- **Passport.js + JWT** — авторизация
- **BullMQ** — очереди задач

**Почему NestJS, а не Next.js API:**
- Бизнес-логика отделена от Next.js → независимая команда backend
- Архитектура из коробки (модули, гварды, интерсепторы)
- Легко вынести в отдельный сервис при росте нагрузки
- Поддержка микросервисов (когда понадобится)

**Альтернатива:** если команда маленькая и хочет простоты — оставить всё в Next.js API routes (Server Actions). Минус: всё в одном процессе, сложно масштабировать отдельно.

#### Real-time сервис
- **Socket.IO** на отдельном Node.js процессе
- Используется для:
  - Чатов
  - Реал-тайм статусов операторов (колл-центр)
  - Уведомлений в браузере

### 1.4 База данных

#### Основная БД
- **PostgreSQL 16+**
- **Multi-tenancy:** schema-per-tenant
  - Каждая компания-клиент = отдельная PostgreSQL schema
  - Общая схема `public` для tenant-registry, биллинга
- **Расширения:**
  - `pgcrypto` — шифрование sensitive полей
  - `pg_trgm` — full-text поиск
  - `uuid-ossp` — UUID-генерация

**Хостинг БД:**
- **MVP (Vercel этап):** Neon (serverless PostgreSQL, branch-friendly) или Supabase
- **Production (Узбекистан):** self-hosted PostgreSQL на VPS / dedicated server

#### Кэш и сессии
- **Redis 7+**
- Используется для:
  - Кэширование тяжёлых запросов
  - Хранение сессий
  - Pub/Sub для Socket.IO
  - Очереди BullMQ
  - Rate limiting
- **MVP:** Upstash Redis (serverless)
- **Production:** self-hosted Redis

#### Аналитика
- **ClickHouse** (для Этапа 3+) — для KPI, отчётов, агрегаций по миллионам записей
- В MVP — обычные SQL-запросы к PostgreSQL с агрегацией

### 1.5 Хранилище файлов

- **MVP:** Vercel Blob или Cloudflare R2 (S3-совместимое, дешёво, без egress fees)
- **Production:** локальное S3-совместимое решение в Узбекистане (MinIO self-hosted)

**Что хранится:**
- Документы сотрудников (договоры, паспорта, дипломы)
- Аватары
- Прикрепления к задачам и чатам

### 2.3 Multi-tenant data flow

```
1. Запрос приходит:    GET /api/employees
                       Host: acmebank.hrcrm.uz
                       Auth: Bearer JWT_TOKEN

2. Middleware:
   - Извлекает поддомен → tenant_slug = "acmebank"
   - Валидирует JWT → user_id, tenant_id из токена
   - Проверка: совпадает ли tenant в URL и в токене?

3. TenantContext устанавливает:
   - SET search_path TO tenant_acmebank, public;

4. Все Prisma-запросы автоматически работают в schema "tenant_acmebank"

5. Ответ возвращается → данные ТОЛЬКО этой компании
```

## 3. Дизайн базы данных

### 3.1 Подход к multi-tenancy

**Схема:** schema-per-tenant в одной PostgreSQL базе.

```
PostgreSQL Database "hrcrm"
│
├── schema "public"                 # Общая для платформы
│   ├── tenants                     # Реестр компаний-клиентов
│   ├── subscriptions               # Тарифы и подписки
│   ├── billing_transactions        # История платежей
│   ├── platform_admins             # Супер-админы
│   └── platform_audit              # Логи действий супер-админов
│
├── schema "tenant_acmebank"        # Данные компании "Acme Bank"
│   ├── users
│   ├── departments
│   ├── time_entries
│   ├── tasks
│   └── ... (все таблицы клиента)
│
└── schema "tenant_grandhotel"      # Данные компании "Grand Hotel"
    ├── users
    ├── departments
    └── ...
```

### 3.2 Общие таблицы (schema "public")

```sql
-- Реестр компаний-клиентов
CREATE TABLE tenants (
    id            UUID PRIMARY KEY DEFAULT uuid_generate_v4(),
    slug          VARCHAR(63) UNIQUE NOT NULL,  -- для поддомена
    name          VARCHAR(255) NOT NULL,
    industry      VARCHAR(50),                  -- it/banking/callcenter/horeca
    country       VARCHAR(2) DEFAULT 'UZ',
    timezone      VARCHAR(50) DEFAULT 'Asia/Tashkent',
    status        VARCHAR(20) DEFAULT 'trial',  -- trial/active/suspended/cancelled
    schema_name   VARCHAR(63) UNIQUE NOT NULL,  -- "tenant_acmebank"
    created_at    TIMESTAMPTZ DEFAULT NOW(),
    trial_ends_at TIMESTAMPTZ,
    metadata      JSONB
);

CREATE TABLE subscriptions (
    id            UUID PRIMARY KEY DEFAULT uuid_generate_v4(),
    tenant_id     UUID REFERENCES tenants(id),
    plan          VARCHAR(50),                  -- starter/business/enterprise
    employees_limit INT,
    modules       JSONB,                        -- {"it": true, "horeca": false, ...}
    price_amount  DECIMAL(10,2),
    price_currency VARCHAR(3) DEFAULT 'UZS',
    billing_period VARCHAR(20),                 -- monthly/yearly
    started_at    TIMESTAMPTZ,
    expires_at    TIMESTAMPTZ
);

CREATE TABLE billing_transactions (
    id            UUID PRIMARY KEY DEFAULT uuid_generate_v4(),
    tenant_id     UUID REFERENCES tenants(id),
    amount        DECIMAL(10,2),
    currency      VARCHAR(3),
    provider      VARCHAR(50),                  -- click/payme/stripe
    provider_id   VARCHAR(255),
    status        VARCHAR(20),                  -- pending/success/failed/refunded
    created_at    TIMESTAMPTZ DEFAULT NOW()
);
```

### 3.3 Таблицы клиента (schema "tenant_*")

```sql
-- Пользователи (сотрудники компании)
CREATE TABLE users (
    id            UUID PRIMARY KEY DEFAULT uuid_generate_v4(),

    phone         VARCHAR(20),
    password_hash TEXT NOT NULL,

    avatar_url    TEXT,
    department_id UUID REFERENCES departments(id),
    position      VARCHAR(255),

    timezone      VARCHAR(50),

    hired_at      DATE,
    terminated_at DATE,
    status        VARCHAR(20) DEFAULT 'active', -- active/inactive/terminated

    created_at    TIMESTAMPTZ DEFAULT NOW()

CREATE TABLE time_entries (

    ended_at      TIMESTAMPTZ,
    type          VARCHAR(20) DEFAULT 'work', -- work/lunch/break
    start_location POINT,                      -- геолокация при начале
    end_location  POINT,
    start_ip      INET,
    end_ip        INET,

CREATE INDEX idx_time_entries_user_date ON time_entries(user_id, started_at);

-- Задачи
CREATE TABLE tasks (
    id            UUID PRIMARY KEY DEFAULT uuid_generate_v4(),
    title         VARCHAR(500) NOT NULL,
    description   TEXT,
    status        VARCHAR(20) DEFAULT 'new', -- new/in_progress/done/rejected/overdue
    priority      VARCHAR(20) DEFAULT 'medium', -- low/medium/high/urgent
    creator_id    UUID REFERENCES users(id),
    deadline      TIMESTAMPTZ,
    completed_at  TIMESTAMPTZ,
    created_at    TIMESTAMPTZ DEFAULT NOW(),
    updated_at    TIMESTAMPTZ DEFAULT NOW()
);

CREATE TABLE task_assignees (
    task_id       UUID REFERENCES tasks(id) ON DELETE CASCADE,
    user_id       UUID REFERENCES users(id),
    PRIMARY KEY (task_id, user_id)
);

CREATE TABLE task_comments (
    id            UUID PRIMARY KEY DEFAULT uuid_generate_v4(),
    task_id       UUID REFERENCES tasks(id) ON DELETE CASCADE,
    author_id     UUID REFERENCES users(id),
    content       TEXT NOT NULL,
    created_at    TIMESTAMPTZ DEFAULT NOW()
);

CREATE INDEX idx_tasks_status ON tasks(status);
CREATE INDEX idx_tasks_deadline ON tasks(deadline);

-- Заявки (отпуска, больничные)
CREATE TABLE requests (
    id            UUID PRIMARY KEY DEFAULT uuid_generate_v4(),
    user_id       UUID REFERENCES users(id),
    type          VARCHAR(50),                -- vacation/sick/dayoff/other
    start_date    DATE NOT NULL,
    end_date      DATE NOT NULL,
    reason        TEXT,
    status        VARCHAR(20) DEFAULT 'pending', -- pending/approved/rejected
    approver_id   UUID REFERENCES users(id),
    approver_comment TEXT,
    decided_at    TIMESTAMPTZ,
    created_at    TIMESTAMPTZ DEFAULT NOW()
);

-- Документы сотрудника
CREATE TABLE documents (
    id            UUID PRIMARY KEY DEFAULT uuid_generate_v4(),
    user_id       UUID REFERENCES users(id),
    type          VARCHAR(50),                -- contract/passport/diploma/medical
    name          VARCHAR(255),
    file_url      TEXT,
    file_size     INT,
    expires_at    DATE,                       -- срок действия
    uploaded_by   UUID REFERENCES users(id),
    created_at    TIMESTAMPTZ DEFAULT NOW()
);

-- Чаты
CREATE TABLE chats (
    id            UUID PRIMARY KEY DEFAULT uuid_generate_v4(),
    type          VARCHAR(20),                -- direct/department/group
    name          VARCHAR(255),
    department_id UUID REFERENCES departments(id),
    created_at    TIMESTAMPTZ DEFAULT NOW()
);

CREATE TABLE chat_members (
    chat_id       UUID REFERENCES chats(id) ON DELETE CASCADE,
    user_id       UUID REFERENCES users(id),
    joined_at     TIMESTAMPTZ DEFAULT NOW(),
    PRIMARY KEY (chat_id, user_id)
);

CREATE TABLE messages (
    id            UUID PRIMARY KEY DEFAULT uuid_generate_v4(),
    chat_id       UUID REFERENCES chats(id) ON DELETE CASCADE,
    author_id     UUID REFERENCES users(id),
    content       TEXT,
    attachments   JSONB,
    reply_to_id   UUID REFERENCES messages(id),
    created_at    TIMESTAMPTZ DEFAULT NOW(),
    edited_at     TIMESTAMPTZ
);

CREATE INDEX idx_messages_chat_date ON messages(chat_id, created_at);

-- Уведомления
CREATE TABLE notifications (
    id            UUID PRIMARY KEY DEFAULT uuid_generate_v4(),
    user_id       UUID REFERENCES users(id),
    type          VARCHAR(50),
    title         VARCHAR(255),
    body          TEXT,
    data          JSONB,
    read_at       TIMESTAMPTZ,
    created_at    TIMESTAMPTZ DEFAULT NOW()
);

-- Журнал аудита
CREATE TABLE audit_log (
    id            BIGSERIAL PRIMARY KEY,
    user_id       UUID,
    action        VARCHAR(100),               -- "user.login", "task.create", etc.
    entity_type   VARCHAR(50),
    entity_id     UUID,
    changes       JSONB,
    ip_address    INET,
    user_agent    TEXT,
    created_at    TIMESTAMPTZ DEFAULT NOW()
);

CREATE INDEX idx_audit_user ON audit_log(user_id, created_at);
CREATE INDEX idx_audit_entity ON audit_log(entity_type, entity_id);
```

## 4. API дизайн

### 4.1 Принципы

- **REST + JSON** для основных операций
- **WebSocket** для real-time (чаты, статусы)
- **Версионирование** в URL: `/api/v1/...`
- **Multi-tenant routing** по поддомену: `acmebank.hrcrm.uz/api/v1/...`
- **OpenAPI 3.1 спека** автогенерируется из NestJS (Swagger)
- **JWT** для авторизации (access token 15 минут + refresh token 30 дней)

### 4.13 Стандарты

- Все timestamps в **ISO 8601** с timezone
- Пагинация: `?page=1&limit=20` → ответ содержит `meta.pagination`
- Фильтрация: `?status=active&department=hr`
- Сортировка: `?sort=-created_at` (минус = desc)
- Rate limiting: 100 запросов/мин на пользователя
- CORS: разрешён только для домена клиента
